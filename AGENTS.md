# AGENTS.md

Guidance for coding agents in this repository. `CLAUDE.md` imports this file.

## What this is

**Corpus** — a Git-free **canonical markdown context store for teams**:
non-engineers write markdown documents, group them into ordered **agent
collections**, and agents consume those collections over **MCP** (OAuth
bearer). Multi-tenant — **Organization → Project** (one default Project
materialized per Org, hidden until a second exists). A central D1
control plane (identity, organizations, projects, membership) plus one
per-Project `ProjectStore` Durable Object holding that project's
documents, collections, and folders in SQLite. Web UI is TanStack Start.

"Workspace" is a reserved word — no row, type, route, or id. Tenancy is
Org / Project only.

## Hard rules

- **Never** access a `ProjectStore` other than via `storeFor(env, projectId)`
  (`src/control/store-for.ts`). It is the multi-tenant boundary.
- **Never** hand-edit `src/control/schema/better-auth.ts` (generated by
  `pnpm auth:schema`) or re-define the bundle shape in
  `src/store/domain/bundle.ts` (the bundle is a cross-product contract
  maintained outside this repo — implement it, do not redefine).
- **Never** open a transaction outside `ProjectStore.write()`. Atomicity
  is `ctx.storage.transaction` on the DO's SQLite; write use-cases live
  inside the DO.
- **Never** use `useEffect` for data fetching. Loaders + server fns
  only.
- **Never** read AppErrors with a message regex; classify on
  `instanceof` / `kind` / library-provided structured codes.
- **Always** construct branded ids via the matching `as*` constructor
  at a trust boundary, never with `as` casts elsewhere.
- **Always** owner-gate Connection administration (create / rename /
  delete / bind a new OAuth credential / mint API key) with
  `requireProjectOwner` from `src/lib/server/shared.ts` or an equivalent
  resolved-Connection owner check. Connection-row writes are scoped by
  `(connectionId, projectId)`.
- **Always** owner-gate bundle export and import with
  `requireProjectOwner`; the bundle is a project owner artifact, never a
  Connection/MCP surface.

## Verification gate

Every change must pass `pnpm check` — the single gate CI also runs:
`pnpm typecheck`, `pnpm lint`, `pnpm format:check`,
`pnpm check:do-migrations`, `pnpm check:event-log-migrations`,
`pnpm test` (full suite — vitest-pool-workers, real DO + D1), and
`pnpm build`, in that order. `pnpm check` is read-only. Use
`pnpm check:fix` to auto-fix formatting + lint (`prettier --write`,
`eslint --fix`) before running the gate. The gate is defined once in
`package.json`; do not re-list its steps in CI or docs — reference
`pnpm check`.

The migration checks run unconditionally as part of `pnpm check`, but
the regeneration remedies are conditional: when `src/db.ts` changes,
`pnpm db:generate:do`; when `src/event-log-db.ts` changes,
`pnpm db:generate:event-log`.

First-time setup: `pnpm db:migrate`. A stale local D1 fails with
`table account already exists` — `rm -rf .wrangler/state/v3/d1` and
re-run.

## Architecture

Two planes. **Control plane** = D1 (`src/control/*`), central across
orgs / projects. **Data plane** = per-Project `ProjectStore` SQLite. A
Collection is an entity _inside_ the DO, never a partition key —
Documents are shared across Collections via `includes`; the whole
graph lives in one DO per Project, written in one transaction.

Tenancy: `organization` → `project` → `{Documents, Collections}`.
`organization`, `member`, `invitation` are owned by the Better Auth
organization plugin — not hand-rolled. Nicia owns only `project`,
FK'd to `organization.id`. Membership is the Better Auth `member`
table (`owner | member` role). Org creation goes through
`auth.api.createOrganization`; the plugin's `afterCreateOrganization`
hook materializes the default project, and `afterRemoveMember` /
`afterUpdateMemberRole` bump `project.authEpoch` (prompt revoke). The
plugin seam: `src/control/org-lifecycle.ts`.

Entry points to read first:

1. `src/server.ts` — Worker entry; routes `/api/auth/*`, `/mcp`,
   `/.well-known/*`, `/healthz` to the Hono app (`src/api.ts`),
   everything else to TanStack Start. `src/start.tsx` opts out of
   TanStack's default CSRF middleware and re-adds an explicit
   same-origin check.
2. `src/project-store.ts` — the `ProjectStore` DO; orchestration only,
   the only place a transaction opens.
3. `src/lib/server/*.ts` — TanStack Start server functions, one file
   per feature. Routes import from the specific file (no barrel).
   `shared.ts` holds `storeOf` / `authedUserId` / `requireProjectOwner`.

## Data-plane layering

Four layers; keep new code in its layer.

- **Transport** (`lib/server/*.ts`, `mcp.ts`, `api.ts`): Zod-validate
  external input, construct branded ids, throw `AppError` at the
  boundary, map DO results → plain DTOs. No business logic.
- **Application** = the `ProjectStore` DO: orchestration + unit-of-work
  owner. Every transport reuses behavior by calling the **same DO
  method**. `mcp.ts` depends on the narrow `McpExecutor` port, not on
  `ProjectStore`; widen via more ports, not by importing the class.
- **Repositories** (`src/store/repos/`): all Drizzle/TypeGraph access.
  Constructed per call from the active `GraphHandle` / `LedgerDb` via
  the DO's `unit()` — correct inside and outside a transaction.
  Methods are domain operations (`attach → AttachResult`), not 1:1
  passthroughs.
- **Domain** (`src/store/domain/`): pure, zero-IO — conflict math,
  change-event vocabulary, version-node constructors, chain verifier,
  retention selection, bundle types. Unit-tested without a DO. New
  event / version rules join here, not as literals in the DO.

The atomic multi-store write (graph + ledger in one tx) stays
**visible** in the DO — that coupling is the DO's defining
responsibility.

## Web data layer (TanStack Start)

- Route loaders call server fns; reads go through `Route.loader` +
  `Route.useLoaderData()`. Mutations call a server fn then
  `useRouter().invalidate()`. Auth gating is `redirect()` from a
  loader. Re-seed-on-change is a version-keyed remount
  (`<Editor key={doc.docVersion} … />`), not an effect.
- Server functions must return plain JSON-serializable shapes. DO RPC
  returns carry non-serializable branding that fails TanStack's
  `ValidateSerializableMapped`; every handler maps DO results into
  explicit plain types (`DocMeta`, `ColDetail`, …) with explicit
  `Promise<…>` return annotations.
- Keep browser-imported server-function modules free of server-only dependency
  graphs. When a feature needs auth, environment, or email implementation,
  expose the `createServerFn` wrappers from `feature.functions.ts` and put the
  implementation in `feature.server.ts`; routes import the wrapper module.
- Request middleware (`sessionRequestMiddleware`) runs once per HTTP
  request; function middleware (`authMiddleware`, `projectMiddleware`)
  runs once per server-fn invocation. Parallelize independent loader
  calls with `Promise.all`; collapse multi-call loaders into one
  server fn where practical.
- `assertServerContext(context)` (in `src/lib/server-context.ts`) is
  the single guard that narrows the injected context — never
  re-implement.

## Auth

- Better Auth + `@better-auth/oauth-provider` + the core
  `organization` plugin (`better-auth/plugins`); versions live in
  `package.json`. Use the OAuth Provider for MCP bearer — **not** the
  deprecated
  `better-auth/plugins/mcp`. Pair with `jwt()`.
- The Better Auth Drizzle schema is generated. `pnpm auth:schema`
  runs `auth generate` against `src/control/auth.cli.ts` and writes
  `src/control/schema/better-auth.ts` verbatim (lint/format-ignored).
  `auth.cli.ts` must mirror `src/auth.server.ts`'s plugin surface
  (schema-affecting options only). Regenerate on any Better Auth bump,
  then `pnpm db:generate`. `src/control/schema/app.ts` holds only the
  Nicia-owned tables (`project`, `connection`, `api_key`, seams).
- **Connection administration is owner-only** (create / rename /
  delete / bind a new credential / mint API key); reads via existing
  credentials are role-agnostic. Admin SQL is scoped by
  `(connectionId, projectId)` — a stray id from another project is a
  silent no-op. Regression keeper:
  `test/connections-cross-tenant.test.ts`.
- Invitations: created by the team server fn (not Better Auth's
  `sendInvitationEmail`), `requireEmailVerificationOnInvitation: false`.
  The invite email sends fail-soft through `src/control/email.server.ts`
  (Cloudflare Email Service or Resend, `EMAIL_PROVIDER`/`EMAIL_FROM` in
  `src/control/env.server.ts`); the `/invite/<id>` link is always shown for
  out-of-band sharing. Better Auth binds acceptance to the invited
  email (intentional, not a bug).
- `BETTER_AUTH_SECRET` is dev-only in `wrangler.jsonc` (`.min(32)` in
  `src/control/env.server.ts`); production sets it via `wrangler secret put`.
- Verify library APIs against the installed version + current docs —
  training data lags these fast-moving deps.
- `pnpm.overrides` pins `kysely` to `0.28.17` — a build workaround, not a
  real dependency (Corpus uses Drizzle/D1, never kysely). kysely 0.29.x
  dropped `DEFAULT_MIGRATION_TABLE` / `DEFAULT_MIGRATION_LOCK_TABLE` from
  its package-root barrel, but `@better-auth/kysely-adapter` still imports
  them from bare `"kysely"`, and better-auth doesn't tree-shake the unused
  adapter — so `pnpm build` breaks even though we never touch kysely
  (`typecheck` passes; the `.d.ts` still declares them). Upstream:
  https://github.com/better-auth/better-auth/issues/9868. Remove the pin
  once better-auth ships a fix; don't float kysely to 0.29+ before then.

## TypeGraph (`@nicia-ai/typegraph`)

- Optimistic concurrency is enforced by the `DocumentVersion`
  node-unique `(slug, docVersion)` constraint. A racing N+1 throws
  `UniquenessError` (`code: "UNIQUENESS_VIOLATION"`) — **not** a SQLite
  message. Classify via `isUniqueViolation` in `src/errors.ts`. The DO
  maps it to a 409 conflict; the whole enlisted tx rolls back together.
  Regression keeper: `test/save-path.test.ts`.
- `kind` is reserved node metadata — use a domain-specific prop name.
- `Node` / `Edge` shapes are flat (not under `.props`).
- Look entities up with `find({ where })`, not `findByConstraint`.

## Durable Object migrations

`wrangler.jsonc` has the initial migrations for `ProjectStore` and
`EventLogStore`. Once deployed, **append** new migration entries —
never edit a deployed one.

DO SQLite ledger schemas are applied by the Drizzle DO migrator at DO
init from `drizzle-do/` and `drizzle-event-log/` (source: the
`sqliteTable` defs in `src/db.ts` and `src/event-log-db.ts`).
drizzle-kit owns this DDL; no hand-written DDL. Baselines are
regenerated, not migrated forward — safe only because there is no
production data. TypeGraph bootstraps its own node / edge tables and
materializes the indexes declared in `canonicalGraph` from `ensureStore`.
Physical upgrades to TypeGraph-owned indexes that its idempotent bootstrap
cannot replace also run there, guarded by a Durable Object storage marker;
do not model them as a Drizzle or `wrangler.jsonc` migration. Regression
coverage lives in `test/do-ledger-migrator.test.ts`. `drizzle/`,
`drizzle-do/`, `drizzle-event-log/` are prettier-ignored so `pnpm format`
can't desync the byte-for-byte check.

## Retention

`project.policy` is JSON, parsed defensively:
`{ retention: { documentVersionDays?, changeEventDays?, blobDays? } }`;
absent / unparseable = "forever". `reconcileRetention(env)` loads each
live project's policy and calls `ProjectStore.reapExpired(retention)`
in one atomic tx. Invariants the reap upholds:

- A document's head version and any version still pinned by a live
  `CollectionVersion` are never reaped.
- A blob is deleted only when no surviving `Document` /
  `DocumentVersion` references its hash.
- The verifier treats a pruned-ancestor gap as expected (asserts
  parent link only across contiguous present versions, null-genesis
  rule only at true `docVersion === 1`).

`reapExpired` takes a test-only `nowMs` clock seam.

## Bundle (the alignment contract)

The portable bundle is a cross-product contract maintained as a
canonical spec outside this public repository — **implement it, never
re-define it here**. `exportBundle` serializes deterministically;
`importBundle` recomputes `integrity.rootHash` and rejects on mismatch
**before** any write, then reconstructs in one atomic tx in dependency
order. Owner-only; reachable via the `exportBundle` / `importBundle`
server fns (Settings → Export in the UI). Deliberately not on MCP —
this is an owner artifact, never an agent credential's path. The
contract test (`test/bundle.test.ts`) asserts export → import → export
is byte-identical (modulo `manifest.exportedAt`); **if it fails, fix
the model, not the test**.

## Testing

`vitest.config.ts` uses vitest-pool-workers with real Worker bindings,
`isolatedStorage: false`. D1 migrations applied via
`test/apply-migrations.ts`. `test/_helpers.ts` carries a per-module
`SALT` so user emails don't collide across test files sharing D1.

## TypeScript conventions

ESLint enforces the basics (typescript-eslint strict + stylistic,
import-x, react-hooks, functional / `type-declaration-immutability`,
vitest). Don't relitigate. `pnpm lint` exit 0; `pnpm format:check`
clean.

Reviewer-upheld rules (not lint-enforced):

- Validate external input (HTTP, MCP, server-fn) with Zod at the point
  of ingestion. Use `safeParse` at trust boundaries, not type
  assertions.
- Under `exactOptionalPropertyTypes`: assemble optional-field objects
  with `compact()` from `src/util.ts`; use `T | undefined` for optional
  properties fed Zod-inferred values.
- `Readonly<{…}>` for value / DTO / domain types. Discriminated unions
  for state. `as const` for literals. Concrete return types on every
  exported function.
- Prefer `undefined` over `null` in domain and return types (the
  SQL/JSON boundary is the exception — see below).
- Object parameters for functions with repeated same-typed fields.
- Named exports only. Co-locate types with their implementation;
  export those that are public API.
- Application errors extend `AppError` (`src/errors.ts`) with a `kind`
  from the taxonomy (`validation | unauthorized | forbidden |
not_found | conflict | internal`) and `this.name` set. Throw at
  framework boundaries; return discriminated-union results across
  internal seams.
- Branded ids (`src/ids.ts`): a raw `string` is not interchangeable
  with an id. Construct via the matching `as*` constructor at a
  boundary (DB read, Zod-validated input, auth session). Brands
  survive the DO RPC stub. Organization / project _slug_ stays plain
  `string`; routing keys on `project.id`.
- Naming: PascalCase types, camelCase values/props, SCREAMING_SNAKE
  constants, kebab-case filenames. Routes under
  `src/routes/p/$projectId/` use the directory-tree convention
  (`route.tsx` for layouts); top-level routes use the dotted
  convention.

Deliberate exceptions — do not "fix" these:

- `null` at the SQL/JSON boundary (Drizzle nullable columns, JSON
  encoded into `change_events.beforeJson` / `afterJson`) is fine; the
  "prefer `undefined`" rule applies to domain/return types only.
- `src/control/env.server.ts` uses Zod `.parse` (throws) at startup —
  fail-fast on a missing/weak secret is intended.
- `src/server.ts` keeps a `default` export (Workers runtime entry).
  `interface Register` in `server.ts` / `router.tsx` stays `interface`
  (TanStack declaration merging requires it).
- `asLedgerDb()` in `src/project-store.ts` and the `as*` brand
  constructors in `src/ids.ts` are the only sanctioned casts.
- DTOs in `src/lib/server/*.ts` intentionally mirror DO return shapes;
  cross-cutting ones (`DocMeta`, `ColMeta`, `Attachment`) co-locate
  with their feature file and are imported by `session.ts` (acyclic).
- Only `control/schema/index.ts` is a sanctioned barrel; broad
  `index.ts` re-exports are rejected (tree-shaking, cycle risk).

## Design context

UI work starts with `PRODUCT.md` (strategic: register, users, purpose,
personality, anti-references, design principles, accessibility) and
`DESIGN.md` (visual: palette, type, spacing, layout, motion, decisions
log), both at the repo root. Read them before touching any surface in
`src/routes/` or `src/components/`.

- **Register:** `product` — design serves the product. Marketing
  (corpus-site) is a separate repo; do not import its lighter blue-500
  accent into the app.
- **Five strategic principles** (PRODUCT.md): the system recedes so the
  graph speaks · show the truth, not the tool · never lose a write ·
  approve, then serve · one source, many readers.
- **Visual discipline (DESIGN.md):** slate neutral ramp + exactly ONE
  accent (`#2563eb` blue-600), spent on primary action and the
  shared-linkage fan-out only. Review-state colors (amber/green/rose)
  and code-syntax colors are scoped exceptions, not new accents.
  Minimal-functional motion (cap 250ms; graph never animates layout).
  Geist (self-hosted) throughout; tabular-nums on all numeric data.
- **Anti-references (PRODUCT.md):** no SaaS marketing dashboards, no
  Notion-style decorative chrome, no AI-default cream/warm-neutral
  body, no generic Linear-clone surface (dark-by-default / glass /
  kinetic motion).
- **Accessibility:** target WCAG 2.1 AA. Review states must not rely on
  hue alone — pair every wash with a text/icon cue. `prefers-reduced-motion`
  honored (motion is minimal by default).
